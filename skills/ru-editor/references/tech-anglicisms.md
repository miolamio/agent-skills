# Технические англицизмы в русском тексте

## Принцип

Если у английского термина есть устоявшийся русский эквивалент, использовать русский. Английский оставлять только когда: (а) русского эквивалента нет, (б) русский вариант длиннее в 3+ раза и неудобен, (в) аудитория — узкие специалисты, для которых английский термин однозначнее.

---

## Типы проблем

### 1. Буквальные кальки (calques)

Дословный перевод английского выражения, звучащий неестественно по-русски.

| Калька | Оригинал | Естественный русский |
|--------|----------|---------------------|
| высокорычажная техника | high-leverage technique | самая эффективная техника |
| низко висящий фрукт | low-hanging fruit | простая задача |
| болевая точка | pain point | проблема, слабое место |
| дорожная карта | road map | план |
| узкое горлышко | bottleneck | ограничение, затор |
| серебряная пуля | silver bullet | универсальное решение |

**Правило:** если выражение — метафора из чужого языка и звучит странно при буквальном прочтении, это калька.

### 2. Транслитерированные англицизмы

Английские слова, записанные кириллицей, при наличии русского эквивалента.

| Англицизм | Русский эквивалент | Когда оставить |
|-----------|-------------------|----------------|
| хелперы | вспомогательные функции | никогда |
| дебаг | отладка | никогда |
| фикс | исправление | в dev-чатах допустимо |
| тулинг | инструменты | никогда |
| воркфлоу | рабочий процесс | в узком dev-контексте допустимо |
| чеклист | контрольный список | в dev-контексте допустимо |
| кастомный | пользовательский | в dev-контексте допустимо |
| краш | сбой, падение | никогда |
| билд | сборка | в dev-контексте допустимо |
| деплой | развёртывание | в dev-контексте допустимо |
| ивент | событие | никогда в доках |
| тосты (UI) | всплывающие уведомления | пояснить при первом использовании |

**Правило:** если слово можно заменить одним-двумя русскими словами без потери смысла и аудитория — не узкий dev-чат, заменить.

### 3. Непереведённый английский в русском тексте

Английские термины, оставленные без перевода в русскоязычном тексте.

| Английский | Русский |
|-----------|---------|
| edge cases | граничные случаи |
| error boundaries | обработчики ошибок |
| accessibility | доступность |
| best practices | лучшие практики, проверенные подходы |
| trade-off | компромисс |
| bottleneck | узкое место |
| use case | сценарий использования |
| proof of concept | прототип, проверка идеи |
| stakeholder | заинтересованная сторона |
| onboarding | адаптация, знакомство |

**Правило:** в презентациях и документации для широкой аудитории переводить. В коде и CLI-примерах — оставлять как есть (это имена, а не текст).

### 4. Смешанные конструкции (англо-русские гибриды)

Английское слово + русское окончание/суффикс, или наоборот.

| Гибрид | Исправление |
|--------|------------|
| accessibility-лейблы | атрибуты доступности |
| error-хендлинг | обработка ошибок |
| кэш-инвалидация | сброс кэша |
| юзер-френдли | удобный для пользователя |
| тест-кейсы | тестовые сценарии, проверочные примеры |

**Правило:** если обе части можно сказать по-русски — сказать по-русски.

### 5. Жаргон без пояснения

Профессиональный термин, понятный специалистам, но непрозрачный для целевой аудитории.

**Решение:** при первом упоминании добавить пояснение в скобках.

| Жаргон | С пояснением |
|--------|-------------|
| тосты | тосты (всплывающие уведомления) |
| спиннеры | спиннеры (индикаторы загрузки) |
| мок | мок (заглушка для тестов) |
| линтер | линтер (проверка стиля кода) |

---

## Исключения: когда НЕ переводить

1. **Имена собственные и бренды:** Claude Code, Git, GitHub, npm, React.
2. **Команды и код:** `/commit`, `npm run build`, `Ctrl+V` — это синтаксис, не текст.
3. **Устоявшиеся термины без эквивалента:** API, SDK, MCP, PR, CI/CD.
4. **Названия техник-паттернов:** Context Before Task, Constraints-First, Kitchen Sink — если используются как имена собственные (названия анти-паттернов, методов).
5. **Текст для узкой dev-аудитории** в неформальном контексте (Slack, комментарии в коде).
